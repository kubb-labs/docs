# Introducing Kubb v5

Kubb v5 is a ground-up rewrite built around a handful of concepts instead of a long list of features. An adapter parses your spec into a universal AST, plugins walk that AST, parsers turn the result into source code, and storage writes it to disk. The same structure that makes the codebase easier to reason about also makes generation a lot faster.

This post sticks to the concepts and the numbers. For the changes to your config and generated output, go straight to the [migration guide](/docs/5.x/migration).

## Up to 6x faster generation

The [OpenAI spec](https://github.com/openai/openai-openapi) has 242 operations across 2.7 MB of YAML. Generating types, an axios client, Zod schemas, and Faker mocks from it took 14.9 seconds in v4. In v5 it takes 2.5 seconds. Smaller specs land around 2x, and the gap widens as specs grow.

Two changes drive this. In v4, every plugin bootstrapped its own OpenAPI parser, so a config with four plugins parsed the spec four times. In v5, the [adapter](/docs/5.x/guide/concepts/adapters) parses it once and shares the AST with every plugin. On top of that, the new JSX renderer renders synchronously, without the React runtime v4 carried.

The full benchmark tables live in the [migration guide](/docs/5.x/migration#performance).

## One pipeline, five concepts

Everything in v5 hangs off a single layered [pipeline](/docs/5.x/guide/concepts/architecture):

The [adapter](/docs/5.x/guide/concepts/adapters) parses and validates your spec and handles schema naming in one place, with collision-safe names and enum deduplication on by default. The [AST](/docs/5.x/guide/concepts/ast) it produces is spec-neutral and exposes a Babel-style `walk`, `collect`, and `transform` API, which is what lets adapters reach toward AsyncAPI and GraphQL later. [Plugins](/docs/5.x/guide/concepts/plugins) walk that AST and describe files. [Parsers](/docs/5.x/guide/concepts/parsers) turn those files into source code, with `@kubb/parser-ts` for TypeScript and the new `@kubb/parser-md` for Markdown straight from your spec. [Storage](/docs/5.x/guide/concepts/storage) writes the result to disk.

You wire none of this up yourself. `defineConfig` pre-configures the OpenAPI adapter, the default parsers, and the barrel plugin, so a minimal config only needs `input`, `output`, and the plugins you care about. Every plugin in the suite, from TypeScript and Zod to MSW and TanStack Query, now runs on this pipeline.

## Upgrading

The real changes, from the new `kubb/config` import to per-plugin before/after diffs, live in the [migration guide](/docs/5.x/migration). It opens with an upgrade prompt you can paste into any LLM together with your `kubb.config.ts` to migrate most configurations automatically.

If you hit anything unexpected, open an issue on [GitHub](https://github.com/kubb-labs/kubb/issues).
